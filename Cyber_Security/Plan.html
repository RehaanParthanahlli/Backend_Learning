Phase 02 ★ NEW
Web Dev for Security Pros
Months 3–4

You must understand how web apps are built before you can break or secure them. This is security-focused web dev — not full-stack dev. Learn just enough to understand attack surfaces.

💡 Why Web Dev Matters for AppSec: SQL Injection makes no sense without knowing how databases are queried. XSS makes no sense without knowing how browsers render HTML/JS. Cookie theft makes no sense without understanding sessions. Build first, then break.
🏗️ HTML & the DOM
  • HTML structure — tags, forms, inputs, attributes
  • DOM (Document Object Model) — how browsers parse pages
  • Forms: GET vs POST, form action, hidden fields
  • HTML comments & metadata (often expose info)
  • Security link: XSS, clickjacking, form tampering
🎨 CSS (Basics Only)
  • Selectors, classes, IDs — enough to read pages
  • CSS-based UI redressing (clickjacking attacks)
  • You don't need to design — just read & understand
  • Inline styles vs external stylesheets
  • Security link: CSS injection, content spoofing
⚡ JavaScript — Critical!
  • Variables, functions, loops, conditionals
  • DOM manipulation: document.getElementById, innerHTML
  • Events: onclick, onload, onerror
  • Fetch API / AJAX — async data calls
  • localStorage, sessionStorage, cookies
  • JS in browser console — devtools usage
  • Security link: XSS, CSRF, token theft, prototype pollution
🌐 HTTP Protocol Deep Dive
  • HTTP methods: GET, POST, PUT, DELETE, PATCH, OPTIONS
  • Request/response headers — what each one does
  • Status codes: 200, 301, 302, 400, 401, 403, 404, 500
  • Cookies: Set-Cookie, HttpOnly, Secure, SameSite flags
  • HTTPS vs HTTP — TLS handshake basics
  • HTTP/2 and differences from HTTP/1.1
  • Security link: Session hijacking, cookie attacks, MITM
🗄️ Databases & SQL — Critical!
  • What databases are: tables, rows, columns
  • Basic SQL: SELECT, INSERT, UPDATE, DELETE, WHERE
  • JOINs, UNION, subqueries (core for SQLi)
  • How web apps query databases (parameterized queries)
  • MySQL/PostgreSQL — run queries locally
  • NoSQL basics (MongoDB) — different injection vectors
  • Security link: SQL Injection, NoSQL Injection, data leakage
⚙️ Backend Basics (Node.js)
  • How a web server works (request → logic → response)
  • Node.js Express — you've already done this with Motrix auth system
  • Routes, endpoints, parameters, middleware
  • Reading/writing to a database from code (Supabase client)
  • Environment variables & config files (where secrets hide)
  • Security link: SSRF, command injection, insecure deserialization
🔌 APIs — REST & GraphQL
  • REST API structure — endpoints, methods, JSON
  • Authentication: API keys, Basic Auth, JWT tokens
  • OAuth 2.0 flow — how "Login with Google" works
  • GraphQL — queries, mutations, introspection
  • Using Postman or Insomnia to test APIs
  • Security link: Broken object auth, JWT attacks, GraphQL injection
🔐 Auth & Session Management
  • How login flows work (form → server → session)
  • Password hashing: bcrypt, Argon2 vs MD5 (bad)
  • Session tokens vs JWT tokens — pros/cons
  • Remember-me tokens, password reset flows
  • MFA implementation basics
  • Security link: Broken auth, session fixation, account takeover
🐳 Dev Tools & Environment
  • Browser DevTools: Network tab, Console, Elements
  • Git & GitHub — version control basics
  • Docker basics — containers, Dockerfile, ports
  • VS Code setup for web projects
  • Reading & understanding other people's code
  • Security link: Exposed .git dirs, secrets in repos, container escape
🚫 What you do NOT need to learn: React, Angular, Vue, TypeScript, advanced CSS frameworks, or full-stack app design beyond what you already know. You're learning to read and attack apps, not build production ones. Your existing Node.js + Supabase + Cloudflare stack is more than enough — use it as your attack surface.
🔨 Capstone Mini-Project: You already built this with Motrix! Take your Node.js + Supabase auth system and intentionally break it — try SQL injection on the login form, steal session cookies via XSS, attempt JWT manipulation, bypass RLS policies. This bridges directly into Phase 3 (AppSec). Your real project = your best lab.
